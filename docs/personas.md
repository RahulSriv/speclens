# SpecLens — User Personas

## Persona 1 — The Product Manager (Primary)
- Writes PRDs and user stories daily
- Pain: never sure if the spec is "complete enough" before handing to engineering
- Goal: ship features that match intent, reduce back-and-forth with dev
- Uses: Notion, Confluence, Linear, Google Docs
- Would use SpecLens: before sharing a spec with the team

## Persona 2 — The QA / Test Engineer
- Receives specs to write test cases from
- Pain: finds ambiguities *after* development — filing "bug or feature?" tickets
- Goal: catch edge cases early, write complete test suites
- Would use SpecLens: when a spec lands in their queue

## Persona 3 — The Tech Lead / Senior Engineer
- Reviews specs before sprint planning
- Pain: wastes meeting time asking obvious clarification questions
- Goal: unblock the team faster, avoid mid-sprint surprises
- Would use SpecLens: during sprint grooming / spec review

## Persona 4 — The Solo Builder / Indie Hacker (Open Source Audience)
- Writing specs for their own product
- Pain: no PM or QA to review — gaps surface during build
- Goal: ship tighter, waste less time rebuilding
- Would use SpecLens: before starting any new feature
